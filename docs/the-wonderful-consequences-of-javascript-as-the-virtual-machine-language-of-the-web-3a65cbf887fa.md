# JavaScript 作为网络虚拟机语言的奇妙后果

> 原文：<https://medium.com/hackernoon/the-wonderful-consequences-of-javascript-as-the-virtual-machine-language-of-the-web-3a65cbf887fa>

早在 90 年代末，当<blink>标签还是一个东西，JavaScript 主要用于*提醒(“某事”)*时，我曾与一位外部顾问短暂合作，他以闲聊为生。当时我是一名 Progress 程序员，没有我们今天所知道的互联网，甚至找工作列表都不容易，更不用说找一份新技术的新工作了。这回到了你在纽约时报或 T4 当地报纸上寻找工作的日子。如果你在日常工作中没有使用[技术](https://hackernoon.com/tagged/technology) X，那你自己学会它可不是件小事。</blink>

虽然我喜欢作为一个进步人士的日常生活，但我也设法找到了一些时间去羡慕“克里斯”他在用一些奇异的编程语言工作，在全国飞来飞去，为有趣的公司做有趣的事情。尽管我也在全国各地飞来飞去做着有趣的事情，但我绝对有一种“草总是更绿”的坏毛病。他继续前进，之后很长一段时间我都在做进步的事情。然后，BizTalk 以及最终的 SharePoint 和 life 以其有趣的方式继续发展。但是，我总觉得我错过了什么。Smalltalk 世界似乎充满异国情调，不同寻常，强大，而且…完全遥不可及。进步是奇异的，不寻常的，但一点也不像闲聊。每个人都听说过 Smalltalk，但几乎没有人听说过 Progress。

显然，JavaScript 已经超越了我第一次接触它时的那种警觉。这种情况已经持续了很多年了，尽管对于像我这样的新人来说还是感觉新鲜。我直到三四年前才开始做“真正的”JS 开发，并且毫无理由地抵制了 Angular 整整一年。我通常将 JavaScript 视为辅助工具，用于为 ASP.NET 驱动的网络应用和 SharePoint 添加一些活力。然而，在我终于看到了 Angular 的光芒之后，我开始真正感兴趣了。大约就在那个时候，我冒了险，开始学习 TypeScript。

我接受新东西的速度相对较慢，所以跳到如此新的东西上并不是我的一贯做法。我确实在相对较早的时候采用了 TS，并且我在使用它的过程中度过了一段美好的时光。我喜欢它带来的显而易见的价值——强类型、类、接口以及它允许你遵循标准软件设计模式的自然方式。但我也越来越欣赏它存在的元事实。它本身是一种成熟的语言，可以编译成 JavaScript。它非常接近 JavaScript。如果不了解 TypeScript 应该只是 JavaScript 加上“extra”，就很难开始阅读 TypeScript 但最终，它真的是一种不同的语言。让我们暂时确定一下这个想法。

追溯一点历史。根据我的经验，第一个真正的“一次编写，随处运行”的网络应用程序是以 Java 小程序的形式出现的。小应用程序承诺了这一有趣的功能，你可以编写一些应用程序，而不必太担心桌面操作系统，不管它们是否安装了正确的 TCP/IP 驱动程序，以及你、你的应用程序和你的客户之间的一百万个其他细节。小程序没用。我不知道为什么。然而，JavaScript 最终做到了。我认为，当谷歌发布具有“永远更新”特性的 Chrome 浏览器时，事情已经注定。其他人紧随其后(或者谷歌紧随其后，这并不重要)，这些天来，我们可以假设我们关注的每个浏览器都将能够运行我们的 JavaScript 解决方案。

回到 TypeScript——让我如此感兴趣的一个元事实是，世界上没有任何浏览器能够真正运行 TypeScript。然而，这并不重要，因为它可以编译成 JavaScript，我关心的任何浏览器都可以运行它。

这些天有一些关于 TypeScript 的争论和各种留言板上的问题，类似于“我应该学习 TypeScript 吗？”我个人认为答案是“是”,并且我通常支持这个观点。也就是说，我开始认为这个问题实际上是一种限制，而且过于策略化。战术问题是一个很好的问题，但是如果你退一步，更广泛地思考，这个问题不是“我应该学习 TypeScript 吗？”而是，“我应该学习能够合理地转换成 JavaScript 的[X]语言吗？”

JavaScript 已经成为一种网络机器语言。它无孔不入。你可以相信它无处不在，并相信它能为你的典型应用程序的所有主要用例工作。(让我明确指出，最终，它是在一个安全的盒子中运行的，有许多许多限制，但这并不是技术解决方案的全部。鉴于这一事实，这是 transpilation 的一个很好的目标——例如，将 TypeScript 编译成 JavaScript。好吧，如果你会打字，你就能聊天。事实上，至少有一个 Smalltalk transpiler。确实(又来了！)—实际上已经有很多语言的 transpilers 了。这里只是一个列表:[https://github . com/jashkenas/coffee script/wiki/list-of-languages-that-compile-to-js](https://github.com/jashkenas/coffeescript/wiki/list-of-languages-that-compile-to-js)。

这就是我所说的“美妙的结果”JavaScript 是无处不在的，而且足够强大，你可以把很多编程语言移植到它上面，它们可以在你的目标环境中运行，不会有任何问题。这反过来意味着你有更多有趣的选择来挑选你的编程语言。如果你像我一样，更关心这个职业的艺术和工艺方面，而不是任何具体的最终结果，你可能会很高兴。

它不会是完美的。这些其他语言就其自身价值而言可能好，也可能不好。它们可能是很好的语言，但是不适合翻译。transpiler 的作者可能不擅长这个。毕竟，不是每个人都拥有微软的资源。如果你在一个纯 JavaScript 商店或打字商店，你仍然有潜在的文化障碍需要克服。也就是说，拓展业务从未如此容易。任务跑者喜欢大口和咕噜。支持运行时调试的映射文件和活跃的在线社区消除了我们 15 年前甚至三四年前面临的许多障碍。它当然不是完美的，但它相当不错，我们有充分的理由认为它会变得更好。

这是激动人心和自由的时代。编程世界从未像今天这样充满活力:)。让我们享受并利用这些美妙的结果；).
