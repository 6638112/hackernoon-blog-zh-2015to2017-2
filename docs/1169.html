Firefox(和Tor浏览器)证书锁定漏洞兔子洞的事后分析

原文：https://medium.com/hackernoon/postmortem-of-the-firefox-and-tor-certificate-pinning-vulnerability-rabbit-hole-bd507c1403b4?source=collection_archive---------2-----------------------#2016-09-19

前言

漏洞披露的最初几天是一段非常有趣的旅程。事情开始于 movrcx 发布他对Tor浏览器的攻击(你一定要读一下)。大多数人拒绝了他，因为他们认为他不明白Firefox中的证书锁定是如何工作的，他的设置会通过设计自然地绕过锁定。有趣的是，他们是对的(但仍然大错特错)。走进去， movrcx 对Firefox中的证书锁定是如何工作的没有很好的理解，我也是如此。补充一句，如果他攻击的是Firefox而不是Tor Browser，即使漏洞不存在，他的结果实际上也是一样的。考虑到当时可用的全部信息，很难责怪任何人放弃了他的攻击。我也差点这么做了。

我发现任何奇怪的事情的唯一原因是，我模糊地记得Tor浏览器通过启用更严格的执行级别来强化他们处理锁定的方式。由于这种强化， movrcx 的攻击不应该在Tor浏览器中使用他的设置。这让我更深入地挖掘和探索这个问题，以至于我可以查明问题的根源，然后向 daveaitel 的每日Dave邮件列表提交我自己关于这个主题的帖子，以获得更多关注。在它上线的几个小时内，Tor项目的Nicolas Vigier通过电子邮件向我指出了问题的根源。然后，在整个下午的过程中，我们确定这个问题实际上会在11月的几天内再次出现，然后在12月到1月的 5周内再次出现。在撰写本文时，情况仍然如此，尽管Mozilla发誓要修复他们的流程以防止这种情况发生。一个按时出现的bug非常有趣。

这篇文章的重点是给出关于这种攻击是如何以及为什么进行的详细的技术细节。我的第一篇帖子中有一些猜测，但事实现在已经众所周知，补丁已经开始推出。这将展示事实，而不是所有的猜测(除非另有特别说明)。此外，我将展示这个漏洞不仅会在未来卷土重来，而且更重要的是，它已经在过去几年里在Tor浏览器和Firefox中发生过几次而没有被发现。

证书锁定

证书锁定是强制浏览器在验证与某个域的TLS连接时只使用某些证书的过程。这可以通过浏览器附带的静态证书pin列表来实现，也可以使用一种叫做HTTP公钥密码(HPKP)的标准来实现，这种标准允许一个站点在第一次连接到它时按下它自己的pin。

Mozilla使用证书锁定来保护与addons.mozilla.org(AMO)的连接，这用于大多数火狐扩展的更新。锁定此域的目的是防止恶意CA为AMO生成证书，然后利用该证书对扩展更新过程进行中间人(MiTM)攻击。

HTTP公钥密码(HPKP)在为用户获取证书密码方面有一个“先有鸡还是先有蛋”的问题。该协议是一个第一次使用时信任(豆腐)协议。pin由网站在用户第一次连接到该网站时发送给用户。这意味着第一个连接不受固定保护。更糟糕的是，如果攻击者要中断第一次连接，他们可以按下他们的自己的 HPKP引脚。为了解决这个问题，Chrome和Firefox (IE/Edge不支持HPKP)都在每个版本的浏览器中捆绑了一个HPKP预加载列表。针被包括在那里，以保护任何连接(像第一个网站)，不被推HPKP针保护。一旦一个网站把自己的HPKP标题推给用户，这些pin就会覆盖预加载列表。

你也可以把一个引脚放在这个预加载列表中，即使你包含的站点在第一次连接时没有推送HPKP报头。这些被称为静态引脚，因为它们从未被HPKP覆盖。addons.mozilla.org的就是这种情况。锁定AMO完全依赖于浏览器附带的静态pin。

漏洞

当HPKP标头被推送给用户时，它们会随到期日期一起发送。这是必要的，因为证书有时需要更改，因为它们过期、被泄露、被发现使用不安全的签名或任何其他原因。如果pin没有过期，该网站将无限期地被他们锁定的证书所困。由网站决定到期日期。

这也意味着静态pin需要过期。如果用户不更新他们的浏览器，并且这些pin没有过期，则将来更改其证书的网站将无法从传统浏览器接收连接。这就是为什么当它们过期时停止强制固定也很重要，而不是让它失败。

此处的漏洞是Mozilla未能为静态pin和HPKP预加载列表设置足够长的到期日期，以持续到下一个Firefox版本。让我们具体看一下设置pin到期时间的代码的最后一行。具体文件为2016年7月25日生成的 StaticHPKPins.h ，包含在2016年8月2日发布的Firefox 45.3.0esr的源代码中(截至本文撰写时的当前易受攻击版本，将于2016年9月20日被45.4.0esr替换)。

static const PRTime kPreloadPKPinsExpirationTime = INT64_C(1472903978258000);

到期时间在这里用纪元时间表示。 1472903978258000 此处转换为2016年9月3日星期六11:59:38 GMT 。

回顾一下， StaticHPKPins.h 于2016年7月25日生成。 Firefox 45.3.0esr于2016年8月2日发布。pin码于2016年9月3日到期。Firefox 45.4.0esr要到2016年9月20日才发布。这意味着从 03 Sep 开始，直到用户安装2016年9月20日发布的新版Firefox，静态pin过期且不强制执行。

Tor Browser基于ESR版本，因此在这种情况下，到期日期和缺少强制锁定是相同的。Firefox的常规版本稍微好一点，因为它的pin直到一周后2016年9月10日才过期，这使得它的漏洞期从17天变为10天。

修复

Mozilla有一个自动更新截止日期的流程，并将此问题归咎于该流程失败。

在ArsTechnica的丹·古丁获得的一份声明中，Mozilla发誓要将HPKP加入addons.mozilla.org的行列。他们还承诺，在下一个版本可以延长到期日期之前，确保pin不会过期。以下是他们的声明:

我们对此进行了调查，并将在9月20日星期二的下一个Firefox版本中发布修复程序。我们已经在9月4日修复了开发人员版上的中断自动化问题，但是对于我们的发布和扩展支持发布版本的用户来说，证书锁定已经过期。我们将在addons.mozilla.org服务器上开启HPKP，这样用户一旦访问该网站，即使内置pin码过期，他们仍将受到保护。我们将改变我们的内部流程，以便内置证书pin在未来版本中不会过早过期。

Mozilla似乎已经跟进，并确实已经开始推出versioncheck.addons.mozilla.org的HPKP。

现在推出HPKP有助于Mozilla保护当前版本的Firefox，直到下一个版本于9月20日在T2发布。这不是一个完整的修复，但我个人认为它将风险降低到了一个远低于Firefox威胁模型的水平。

另一方面，Tor有一个非常不同的威胁模型。他们意识到了这一点，并决定将他们为9月20日发布所编译的版本提前4天在9月16日发布。他们的快速反应值得称赞。

旧的就是新的

如果这个bug没有被检查，它会在11月的 返回3天，从12月到1月的 5周内返回。这让我想知道Firefox和Tor浏览器在过去有多容易受到这种攻击。四个月内出现三次对我来说太过分了。

结论和致谢

这是一个非常有趣的追踪bug。它展示了当你处理一个复杂的代码库时，过程的失败是如何的难以发现，并且没有任何迹象表明有什么东西失败了。如果我说运气在这个bug的发现中没有发挥重要作用，那我是在撒谎。如果movrcx在9月3日或9月20日T21之前尝试攻击，他的测试就会失败。只是因为他在这17天的时间内进行了这项研究才被发现。

Mozilla也在研究如何强化扩展更新过程本身，这样即使你将来能够再次破解证书锁定，这样的攻击也不会起作用。该漏洞实际上仅限于通过AMO更新的扩展。遵循自身更新过程的扩展不受影响。Tor浏览器包含的版本中的 HTTPS Everywhere 更新过程没有受到影响，因为它们通过EFF.org进行更新，并在更新下载后进行签名验证。然而，AMO交付的版本可能会像所有其他版本一样受到影响。
