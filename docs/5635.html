…或者只是再次回来

这将是一篇非常个人化的文章，对于那些在过去几年里没有和我一起工作的人来说可能没什么意思。但我仍然想写它，因为我想表达一些现在在我脑海中徘徊的事情。所以…

我要离开 伊纳卡斯 ，我要重新做一名程序员……

走向

在某种程度上，我正在前进。我要离开现在的工作，期待开始新的工作。这是一个新的挑战，在一个新的世界，充满了惊人的发现和新的故事要写。

回去

但在某种程度上，我也在倒退。我将离开我的CTO角色(这是我人生中的第二次),我将再次成为一名程序员。

为什么？

这里有很多为什么的…

为什么我要离开稻香？

有几个原因我不能透露，但没有一个原因足以让我离开一家我帮助建立和成长的公司。在我能找到的例子中，我认为有两个是最重要的:

  • 第一个是不确定性。InakaESI 的项目和工作流程(尤其是在Erlang Solutions 的领域内)是不可预测的。这不是该公司第一次发生这样的事情，但这是第一次这样的情况持续如此长的时间。一年多来，我们一直在与替补人员和项目打交道，这些项目是估计和/或计划的，但从未实现。我不认为这是任何人的错:每个人都意识到了这种情况，并尽最大努力让我们摆脱困境，但我们就是做不到。
  • 第二个更个人化，它与成功有关，或者说与缺乏成功有关。我是一个追求成功的人。我喜欢成功，我一直以此为目标。作为一名程序员，我无数次感到成功，其中许多都是在为伊纳卡斯工作。每当我看到我开发的系统按预期运行时，我都感到很成功。作为一名首席技术官/技术主管，我并不经常有这种感觉。我知道这不完全是我的错，这很大程度上取决于项目、团队、客户、想法等等…但这仍然没有发生。我希望它再次发生。我想再次感受成功。

为什么我愿意再做一次程序员？

嗯……上面的第二点基本上说明了一切。我想建造一些东西，然后看着它工作。我希望能够告诉人们看到了吗……我开发了它！我怀念这种感觉，我知道作为一名开发人员我可以实现这一点。从某种意义上来说，这比做CTO/经理/技术主管更安全。

我讨厌CTO这个角色吗？

我两次离开那个角色，我一定讨厌它，对吗？事实上我不知道。在过去的4年里，我学会了喜欢这个职位。我从伟大的科技领袖那里学到了很多，比如柳文欢·埃伦博根、兰德斯、叶戈尔·布加延科等人。我学到了1:1，如何给予和接受有价值的反馈，如何管理远程团队(以及自己远程工作)，如何不插手并自己解决所有问题，如何领导用我不熟悉的语言编写代码的人，等等…

这不是很受欢迎，但是技术领导谈话是这些知识的结果。我会继续在那里写作吗？谁知道？也许吧？我还有一两件事值得分享。

你还会做CTO吗？

嗯……不太可能，至少在我现在加入的公司不太可能。但你永远不知道。当查德·德普和我以及另外三个人一起创建伊纳卡西的时候，这也不太可能。

再见，谢谢所有的鱼！🐬

5经验教训

在这些公司的这些年里，我学到了很多东西。这些不是前5名，这些是只是5个 …

  1. 沟通是关键:你有没有看过一部电影或肥皂剧，然后想“如果你告诉她这些，我们就可以避免整整三集，你这个笨蛋！”。我总是遇到这种事。在Inaka，我们学到了惨痛的教训。
  2. 指导方针超级重要:有了合适的指导方针，你就不用去想别人是怎么写代码的了。你可以把重点放在他写了什么或者他为什么这么做。这只是好的、一致的、开放的和最新的指导方针的众多好处之一。
  3. 好主意并不代表一切:我们和很多初创公司合作过。我们从他们身上学到，仅仅有一个伟大的想法，即使你能完整地开发它，也不能保证成功。你必须发布它，支持它，监控它，等等…你必须在开始开发过程之前为所有这些活动做好预算。
  4. 评估很难:我们很仔细，这曾经是我们最宝贵的优点之一。我们过去常常会问很多问题，并以高水平的细节进行评估。这太棒了……对于MVP规模的项目，当客户已经知道他们系统的第一次迭代会是什么样的，而我们只是在估计那次迭代。我们最终对大项目使用了同样水平的彻底性，完全不是MVP规模的，并获得了预期的结果。我们了解到:如果你的估算超过了2个月的工作量……你应该估算一些更小的，并把剩余的估算留给未来的迭代。
  5. 教即学:我会让伊娜琪解释的…
